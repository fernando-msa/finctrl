## 2. VULNERABILIDADES E SEGURANÇA

### 2.1 Dependências (npm audit)

- **MODERADO (10):** postcss <8.5.10 (XSS via stringify) — transitivo via next
- **MODERADO:** uuid <11.1.1 (buffer bounds check) — transitivo via firebase-admin
- **Status:** Ambos são transitivos, sem fix direto disponível. Monitorar atualizações.

### 2.2 Código — Achados

#### CRÍTICO: Feedback endpoint sem persistência

- `POST /api/diagnostics/feedback` valida App Check mas **não persiste** o feedback
- Retorna `{ ok: true, payload }` de volta — possível information leak do payload
- **Ação:** Persistir no Firestore OU remover o retorno do payload

#### MÉDIO: Health endpoint sem autenticação

- `GET /api/admin/health` público — aceitável para healthcheck, mas o nome "admin" sugere proteção
- **Ação:** Renomear para `/api/health` ou adicionar auth

#### MÉDIO: Logout não invalida server-side

- Logout apenas limpa o cookie client-side
- Se o token for roubado antes do logout, continua válido até expirar (5 dias)
- **Ação:** Considerar blacklist de tokens ou reduzir TTL

#### BAIXO: X-XSS-Protection deprecated

- Header `X-XSS-Protection: 1; mode=block` é deprecated em navegadores modernos
- **Ação:** Remover, pois CSP é o substituto recomendado (não implementado ainda)

#### BAIXO: CompetenceDate não validada como data

- `competenceDate: z.string().min(7)` aceita qualquer string com 7+ chars
- **Ação:** Validar com regex YYYY-MM

### 2.3 Firestore Rules

- **OK:** Isolamento por `request.auth.uid`
- **OK:** Limite de 40 campos no doc principal, 30 nos subcollections
- **OK:** Delete proibido no doc raiz do usuário
- **MELHORIA:** Subcollection write não valida campos específicos (qualquer key aceita)

### 2.4 Headers de Segurança

- **OK:** HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy
- **FALTA:** Content-Security-Policy (CSP) — importante contra XSS
- **FALTA:** CORS não configurado (API routes aceitam qualquer origin)
